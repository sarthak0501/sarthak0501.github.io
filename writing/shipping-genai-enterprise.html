Shipping GenAI in enterprise:
what actually breaks.

Five lessons from building a production LLM agent on Azure OpenAI — the gap between the prototype that wowed the leadership review and the thing that held up when real engineers tried to break it.

TL;DR

Demos lie. Grounding is the product, not the model. Build evals before you expose users. PII is 30% of the work nobody budgets for. The LLM call is 10% of the system. The interesting engineering lives in the last 10% nobody demos.

An exec opens the customer-health copilot and asks, "how is Contoso doing?" Six seconds later: three bullet points, grounded in eight live data sources, every claim linking back to the ICM, the ADX query, the raw row. Six months earlier that same question took three days and a deck.

The delta between those two worlds — demo script versus real prod — is where most GenAI projects die. Here's what actually broke along the way.

Demos lie.

The prototype that wowed the leadership review was a forty-line script with a clever prompt and three curated questions. It answered them beautifully. Everyone nodded.

Then engineers started using it. One of the first real questions was:

"Which APAC customers with >10TB capacity have both ticket volume trending up and MAL regressions this quarter?"

The prompt melted. The agent confidently cited three customer names. Two didn't exist. One did, but the MAL claim was hallucinated.

The gap between "it works in a demo" and "it holds up in prod" is the gap between a one-hop query over pre-fetched data and a multi-step reasoning problem where the model has to choose tools, compose filters, and stay honest about what it can't see. You won't know your prototype is a toy until someone tries to break it with a real question. That's the point at which you start building the actual system.

Grounding is the product.

Nobody in the org cared whether we used GPT-4 or Claude Opus. They cared that when the agent said "Contoso had three MAL regressions last week," they could click through to the ICMs, the KQL, the raw rows.

One hallucinated customer name and the whole tool loses trust — permanently. That's the asymmetric risk of GenAI in enterprise: a thousand correct answers buy you one user; one confident wrong answer loses you ten. Trust is spent, not earned.

Specific things that moved the number:

  • Chunking telemetry metadata at the metric level, not the document level — the model needed to reason about individual signals, not prose paragraphs
  • Schema-valid outputs for every tool call; reject and retry on malformed JSON rather than letting the model improvise its way out
  • Row-count assertions on every KQL the agent generated — a query returning zero rows would never be surfaced as "I found no regressions"
  • Explicit refusal paths for "I don't have enough context to answer this" — measurably better than confident wrong answers

None of these are exotic. All of them are boring. All of them were the difference between a demo and a tool people actually used.

Evals before users.

We built the eval harness before we exposed the agent to anyone. This was the single highest-leverage decision of the project.

Why: once real users interact, every prompt tweak becomes political. "It was better yesterday" becomes the loudest signal in the room, and the loudest signal is rarely the most important one. You need an objective thing to point at.

The harness was straightforward: 

# golden set: ~180 questions drawn from historical exec queries
# each labeled with the grounded answer + expected citations

for case in golden_set:
    answer = agent.run(case.question)
    scores = {
        "groundedness": grounded_score(answer, case.sources),
        "correctness":  correctness_score(answer, case.expected),
        "citations":    citation_validity(answer.citations),
    }
    harness.log(case.id, prompt_version, scores)

It caught a regression I would never have seen otherwise: a prompt tweak that improved tone but dropped citations in 12% of answers. In a one-off qualitative review nobody would have noticed. The eval flagged it in eight minutes.

A golden set of 180 cases is small enough that a product team can actually maintain it, and large enough to catch real regressions. Start there. Not a thousand synthetic questions — a hundred and eighty real ones.

PII is 30% of the work.

Nobody budgets for this. Everyone gets surprised by it.

Customer names. Ticket contents. Internal ownership. Every data source has its own sensitivity rules, its own approval workflow, its own audit expectations. Getting the right masking in the RAG store, the right redaction in responses, the right logging for who asked what about whom — this was as much code as the agent itself.

The pattern that worked: a middleware layer between the LLM and every tool call that ran PII scrubbing on outputs before they hit the prompt context. Imperfect — regex-based scrubbers always are — but auditable, which is what compliance actually cares about.

Adjacent, underbudgeted work in the same category:

  • Per-user authorization scoping on tool calls — the agent inherits the user's read perms, not the service account's
  • Prompt logging with retention policy — you need the logs to debug, but not forever
  • Refusal for out-of-scope requests — someone always tries to ask your customer-health agent about HR
  • Entra / managed identity plumbing for every downstream service

The LLM call is 10% of the system.

The model call is the cheap part. What took real engineering:

  • Retrieval — hybrid BM25 + embeddings over telemetry metadata, with different chunking strategies per source (a ticket is not a metric definition is not an ownership record)
  • Tool registry — a dozen tools, each with schemas, timeouts, retry policies, cost accounting, and per-tool eval cases
  • Guardrails — input scope classification, output validation, refusal routing, confidence thresholds
  • Observability — every agent trace logged with tool decisions, retries, groundedness scores, latency breakdown
  • Cost controls — token budgets per session, fallback to smaller models for low-stakes paths

If you're scoping a GenAI project around "prompt engineering and maybe some fine-tuning," you are off by 10x. The prompt is the tip of the iceberg. The iceberg is a distributed system with a language model somewhere in the middle.

The interesting engineering lives in the last 10% that nobody puts in a conference talk. The first 90% — frameworks, fine-tuning, the latest retrieval pattern — is mostly a solved problem, or at least a problem the ecosystem is happily solving for you.

The last 10% is where your agent earns its place in production, or loses it.
