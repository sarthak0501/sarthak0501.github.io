// Case 01 · Flagship · GenAI · Microsoft · Azure Storage

MCC AI Agent — an LLM copilot for
"how is this customer really doing?"

A grounded, tool-using agent on Azure OpenAI that replaced 3–5 day exec readouts with real-time natural-language answers over eight live data sources. Shipped with an eval harness, guardrails, and PII-aware prompts so it could hold up under real enterprise load.

Role
Technical lead, design & implementation

Status

Headline

The gap the agent was built to close.

Customer health inside Azure Storage lived across at least eight disconnected tools — telemetry in ADX, tickets in ICM, CRIs in a separate tracker, ownership graphs in yet another service. Every exec readout was a three-to-five day manual hunt: an analyst joined tables in their head, pieced together a narrative, and shipped a deck.

By the time the story was told, the window to act on it was often closed. Critical accounts hit Sev1 before the right person saw the signal.

The core ask from leadership wasn't "build us a chatbot." It was: can anyone in this org, at any hour, ask a plain-English question about a customer and get a trustworthy answer?

How the system is wired.

// agent architecture

At the center: a GPT-4-class agent on Azure OpenAI with function/tool calling. Around it: a retrieval layer that can pull metric definitions, ticket excerpts and ownership records; a tool registry that can execute live KQL against Azure Data Explorer; and a wrapping evaluation + guardrails layer that runs on every response before it ships.

The non-obvious design choice was to treat retrieval as a tool, not a preprocessing step. The agent decides when it needs metric definitions versus raw rows versus ownership lookups, rather than receiving a pre-stuffed context window. That single decision is what let it answer multi-hop questions like "which APAC customers with >10TB capacity have both ticket volume trending up and MAL regressions this quarter?"

The system prompt (redacted).

Every grounded behavior in the agent starts here. The shape of this prompt went through probably 40 revisions — every one scored by the eval harness before it shipped.

// system-prompt.v7redacted · representative
# role
You are a customer-health analyst for Azure Storage. You answer
questions by calling tools against live data. You do not guess.

# grounding rules
- Every numeric claim MUST cite the source (tool + query id).
- If a tool returns zero rows, report that. Do not interpolate.
- If you lack a needed tool or permission, say so and stop.
- Never combine values from different time windows without saying so.

# tool selection policy
1. Look up the customer by TPID before any other call.
2. Prefer aggregated tools (MAL, CRI summaries) over raw KQL.
3. Only write KQL when no aggregated tool can answer the question.
4. Budget: at most 6 tool calls per answer.

# output schema
Return a JSON object with: summary, findings[] (claim, evidence),
confidence (low|med|high), refusal (bool, reason).

Two lines did disproportionate work: "if a tool returns zero rows, report that" and "never combine values from different time windows without saying so." The first killed a whole class of "I found no issues" hallucinations. The second killed a subtler one where the model would happily mix last week's MAL with last quarter's ticket volume.

A KQL query the agent generates.

For the long tail of questions no pre-built tool covered, the agent could write KQL. Every generated query went through a row-count assertion and a syntax check before the results were surfaced.

// agent-generated KQLafter grounding & validation
// resolved from: "APAC customers >10TB w/ MAL regressions this Q"
// agent call: exec_kql(cluster="xstore-analytics", db="...")

StorageAccountMetrics
| where TimeGenerated between (startofquarter(now()) .. now())
| where GeoRegion == "APAC"
| summarize CapacityTB  = avg(CapacityBytes) / 1e12,
            WowMAL      = sum(MALMinutes)
  by TPID, AccountName
| where CapacityTB > 10
| where WowMAL > tolong(prev(WowMAL, 1, 0)) * 1.25
| join kind=inner TicketTrends on TPID
| where TicketsWoWDelta > 0
| project TPID, AccountName, CapacityTB, WowMAL, TicketsWoWDelta
| order by WowMAL desc
| take 25

The answer the user saw wasn't the query. The user saw a paragraph with five customer names and a clickable link to the query, the row count, and the raw table. The query was the receipt — which is what turned "an LLM answered my question" into "a trustworthy signal I can act on."

The eval harness.

Built before any real users touched the agent. A golden set of ~180 historical exec questions, each labeled with its grounded answer and expected citations. Every prompt change ran the full set before ship.

// eval run (abridged)groundedness · correctness · citations
RUN  eval/2026-04-18-prompt-v7  · 180 cases

│ correctness (mean)       │ 0.793  │ 0.821  │ +0.028 │
│ citation validity        │ 0.941  │ 0.827  │ -0.114 │  ← regression
│ refusal (appropriate)    │ 0.876  │ 0.902  │ +0.026 │
│ refusal (over-refused)   │ 0.042  │ 0.058  │ +0.016 │
│ tool-budget exceeded     │ 0.033  │ 0.011  │ -0.022 │
└──────────────────────────┴────────┴────────┴────────┘

VERDICT  block ship — citation validity regressed on 12% of
         cases. investigate §4.2 tone-tweak before merge.

This specific regression is real. A prompt tweak that improved tone also dropped citations on 12% of cases — the kind of thing that would slip past any qualitative review but would have destroyed user trust in production. The harness flagged it in under ten minutes.

Things I'd do again — and one I wouldn't.

Would do again:

  • Build the eval harness before anyone uses the agent. Every hour spent here saved a week of political feedback later.
  • Treat retrieval as a tool, not a preprocessor. Let the model ask for what it needs.
  • Schema-valid outputs everywhere. Reject + retry is boring; it's also what ships.
  • Assert on every KQL the agent writes before surfacing rows.

Would change:

  • We underscoped PII and authorization work by at least 2×. If I were starting today, I'd spec the middleware layer (scrubbing, per-user scoping, audit trail) in week one, not week six.
