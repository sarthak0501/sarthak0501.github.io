// Case 02 · ML Platform · Microsoft · Azure Storage

Account2Vec — turning fleet telemetry into a
reusable ML primitive.

An unsupervised autoencoder that compresses ~100 behavioral metrics per Azure Storage account into a dense embedding — then extends hierarchically to subscriptions and customers. One shared representation powering similarity, anomaly, forecasting, and LLM reasoning across the org.

Role
Designer, model & pipeline owner
Stack
PyTorch · Azure Synapse · ADX / KQL
Scale
M+ accounts embedded at fleet scale
Shape
3-tier: account · subscription · TPID

One feature set, built once.

Every ML effort in Azure Storage — customer similarity, anomaly detection, forecasting, workload segmentation, LLM-based reasoning for the MCC agent — was rebuilding its own feature set from the same raw telemetry. There was no shared representation of what an "account" actually looked like at scale.

This is the shape of wasted ML work inside big orgs. Every team ends up with its own slightly-different version of the same derived features, which means their models can't be compared, their anomaly scores can't be joined, and the next team has to start over. The productivity cost compounds every quarter.

Account2Vec's premise: if ~100 raw metrics describe an account, there's a much smaller latent space that captures the meaningful behavioral variation. Learn that space once, hand the vectors out, let a dozen downstream teams reason in shared coordinates.

Pipeline shape.

// platform architecture

The pipeline runs in four stages: raw XStore telemetry is aggregated into the AccountFeatureMatrix (a stable, versioned view of ~40–50 per-account rolling features); the autoencoder compresses those features into a dense embedding; embeddings are hierarchically pooled into subscription and customer tiers; and the whole surface is instrumented with drift, coverage and staleness monitoring.

The design decision that mattered most was separating the feature matrix from the model. The feature matrix is the platform contract — other teams can use it directly even without embeddings. The embeddings are an optional-but-powerful second layer.

The AccountFeatureMatrix schema.

The single most important artifact in the whole system. If this schema isn't stable, nothing downstream can compound. Versioned, column-level documented, and treated as a product contract.

Every column has a rolling window (7d/30d/90d), is standardized per-region, and is null-masked so newly-created accounts don't drag the embedding space. Boring decisions, enormous downstream leverage.

The training loop.

A standard denoising autoencoder — the interesting engineering is everything around it: batch composition, scheduling, checkpointing, and the hierarchical pool.

// a2v/model.pyrepresentative · abridged
class AccountEncoder(nn.Module):
    def __init__(self, n_features=52, d=32):
        super().__init__()
        self.enc = nn.Sequential(
            nn.Linear(n_features, 128), nn.GELU(), nn.Dropout(0.1),
            nn.Linear(128, 64),         nn.GELU(),
            nn.Linear(64, d),                  # bottleneck = embedding
        )
        self.dec = nn.Sequential(
            nn.Linear(d, 64),               nn.GELU(),
            nn.Linear(64, 128),             nn.GELU(), nn.Dropout(0.1),
            nn.Linear(128, n_features),
        )

    def forward(self, x, mask):
        z = self.enc(x * mask)               # null-mask raw inputs
        xr = self.dec(z)
        return z, xr

# training step (abridged)
for batch in loader:                   # stratified by region + age
    x, mask = batch["feats"], batch["mask"]
    x_noisy = x + noise(x, p=0.15)       # denoising
    z, xr = model(x_noisy, mask)
    loss = masked_mse(xr, x, mask) + \
           0.01 * z.pow(2).mean()        # soft unit-norm
    loss.backward(); opt.step(); opt.zero_grad()

Two details that matter: null-masking the loss (so brand-new accounts with missing history don't pull the embedding toward zero), and a soft unit-norm penalty on the bottleneck (so cosine similarity on the embeddings is meaningful out of the box, which is what downstream teams actually want).

A downstream query — account similarity.

This is the payoff. Once embeddings exist, any team can run a query like "find me 20 accounts similar to this one" without writing a model.

Account → subscription → customer.

Downstream teams don't always reason at the account level. The same platform exposes pooled embeddings at two higher tiers so a forecasting team can ask questions at the TPID level and an anomaly team can operate per-account, without either rebuilding features.

The pooling strategy is a weighted mean with per-axis attention learned against a small set of aggregation tasks — not a plain average, because naive averaging collapses bimodal customers (two very different subscriptions) into a meaningless middle point.

What I'd repeat — and what I'd change.

Would repeat:

  • Separate the feature matrix from the model. The matrix is the contract; the model is an implementation.
  • Version everything. Every schema change ships with a migration plan for downstream consumers.
  • Design docs before code. A good design doc is what turns a model into a platform — it's the thing that names the abstraction for the rest of the org.
  • Instrument drift from day one. You will not have time to add it later, and you will need it.

Would change:

  • I'd invest earlier in the onboarding story for downstream teams. The embeddings are only as useful as the number of teams comfortable reaching for them — that's a docs-and-evangelism problem, not a modeling one.
  • I'd spec the pooling task loss more carefully. Naive averaging was our first pass and cost us a quarter of rework.
